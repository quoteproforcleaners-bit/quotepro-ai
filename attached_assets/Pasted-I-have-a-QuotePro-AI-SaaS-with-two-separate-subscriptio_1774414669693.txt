I have a QuotePro AI SaaS with two separate subscription 
systems that don't talk to each other:
- Web: Stripe Billing (server/routers/integrationsRouter.ts 
  or subscriptions route)
- iOS: RevenueCat (client-side, mobile only)
- Source of truth: users.subscriptionTier in PostgreSQL

I need unified subscription state across all platforms.

1. UNIFIED SUBSCRIPTION TABLE
   Add columns to users table via raw SQL:
   
   stripe_customer_id: text
   stripe_subscription_id: text  
   stripe_subscription_status: text
   revenuecat_user_id: text
   revenuecat_entitlement: text
   subscription_platform: text  -- 'stripe' | 'revenuecat' | null
   subscription_synced_at: timestamp

2. REVENUECAT → STRIPE SYNC
   When a user upgrades on iOS via RevenueCat:
   
   In the route that handles RevenueCat webhooks 
   (or create POST /api/webhooks/revenuecat):
   
   - Verify the webhook authenticity (RevenueCat sends 
     X-Revenuecat-Signature header — verify it)
   - On event type INITIAL_PURCHASE or RENEWAL:
     a) Find user by revenuecat_user_id
     b) Map RevenueCat product ID to QuotePro tier:
        com.quotepro.starter → 'starter'
        com.quotepro.growth → 'growth'  
        com.quotepro.growth.annual → 'growth'
        com.quotepro.pro → 'pro'
        com.quotepro.pro.annual → 'pro'
     c) Update users.subscriptionTier in DB
     d) Update users.subscription_platform = 'revenuecat'
     e) Update users.subscription_synced_at = NOW()
     f) If user doesn't have a Stripe customer record, 
        create one via Stripe API with their email 
        (so they can use the billing portal on web)
   
   - On event type EXPIRATION or CANCELLATION:
     a) Downgrade user to 'free'
     b) Update subscription fields accordingly
     c) Send churn intervention email (see Sprint 6)

3. STRIPE → REVENUECAT SYNC
   When a user upgrades on web via Stripe:
   
   In POST /api/subscription/webhook, after 
   checkout.session.completed:
   
   - Get or create RevenueCat subscriber via REST API:
     POST https://api.revenuecat.com/v1/subscribers/
     {app_user_id}/entitlements/{entitlement_id}/promotional_entitlements
     Authorization: Bearer {REVENUECAT_SECRET_KEY}
   
   - Grant the appropriate entitlement in RevenueCat 
     matching the Stripe tier purchased
   - Set duration to match the Stripe subscription period
   - Update users.subscription_platform = 'stripe'
   - Update users.revenuecat_entitlement in DB

4. GET /api/subscription/status (new unified endpoint)
   Returns the canonical subscription state from DB:
   {
     tier: 'free' | 'starter' | 'growth' | 'pro',
     platform: 'stripe' | 'revenuecat' | null,
     status: 'active' | 'trialing' | 'past_due' | 'cancelled',
     currentPeriodEnd: Date | null,
     isOnTrial: boolean,
     trialDaysRemaining: number | null,
     canManageOnWeb: boolean,  // true if platform is stripe
     canManageOnIOS: boolean   // true if platform is revenuecat
   }
   
   Update useSubscription() hook on both web and mobile 
   to use this endpoint as the source of truth.

5. SUBSCRIPTION CONFLICT RESOLUTION
   If a user somehow has active subscriptions on BOTH 
   platforms (edge case but must be handled):
   
   - Take the HIGHER tier of the two
   - Log a conflict_detected event to analyticsEvents
   - Send an alert email to the admin
   - Do NOT cancel either subscription automatically — 
     require manual review

6. MOBILE SUBSCRIPTION STATUS DISPLAY
   In client/screens/SettingsScreen.tsx (or equivalent):
   
   Add a "Subscription" section that shows:
   - Current plan + price
   - "Subscribed via App Store" or "Subscribed via web"
   - If via App Store: "Manage in iOS Settings →" 
     (deep link to iOS subscription management)
   - If via web: "Manage subscription →" 
     (calls POST /api/subscription/create-portal 
     and opens in browser)
   - If on trial: countdown "X days left in trial"

Show every new route, every schema change, 
the complete RevenueCat webhook handler, 
and the unified status endpoint.
