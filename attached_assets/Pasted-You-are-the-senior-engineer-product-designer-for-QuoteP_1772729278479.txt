A) INVOICE PACKET (QuickBooks-friendly) — NO DIRECT QBO API YET
========================
1. Add a new action button on Quote Details: “Create Invoice Packet”.
2. Invoice Packet contains:
   - Customer info (name, phone, email, address)
   - Service address (if different)
   - Quote ID, date, expiration date
   - Line items with:
     - item_name (e.g., “Deep Clean”, “Standard Clean”, “Inside Oven”)
     - description
     - quantity (default 1)
     - unit_price
     - amount
   - Subtotal, tax (if applicable), total
   - Notes / terms
3. Export formats:
   - PDF “Invoice Packet” (print/email to customer or bookkeeper)
   - CSV “QuickBooks Import Friendly” (line items + customer columns)
   - “Copy invoice text” (clipboard friendly)
4. Keep CSV columns explicit and stable:
   - customer_display_name
   - customer_email
   - customer_phone
   - billing_address_line1
   - billing_city
   - billing_state
   - billing_zip
   - service_date (optional)
   - item_name
   - item_description
   - item_qty
   - item_rate
   - item_amount
   - tax_amount
   - total_amount
   - memo
5. Store Invoice Packet record in DB:
   - invoice_packet_id
   - quote_id
   - created_at
   - pdf_url (or stored blob reference)
   - csv_url (or stored text)
   - status = generated
6. UX:
   - After generation show a modal with: Download PDF, Download CSV, Copy Text, Email to Myself (simple mailto link).
   - Add a short disclaimer: “Designed for easy entry/import into QuickBooks. Not a live sync.”

Implementation notes:
- Use existing auth / userId.
- Ensure PDFs render consistently (avoid layout bugs).
- If file storage exists, use it; if not, store PDF/CSV as generated strings and allow download via endpoint.

========================
C) WEBHOOKS (Zapier/Make bridge) + API KEYS + EVENT LOG
========================
This is the “real integration” layer. Implement outbound webhooks only.

1. Add a new Settings area: “Automations”
   - Generate API Key (rotateable)
   - Webhook URLs list (user can add multiple)
   - Toggle per event type
   - Test webhook button

2. Event types (minimum):
   - lead.created
   - quote.created
   - quote.sent
   - quote.accepted
   - quote.declined
   - invoice_packet.created
   - calendar_stub.created

3. Webhook payload rules:
   - POST JSON
   - Include:
     - event_type
     - event_id (uuid)
     - occurred_at (iso)
     - account_id / user_id
     - data object (lead/quote/invoice/etc.)
   - Include signature header:
     - X-QP-Signature = HMAC_SHA256(api_key, raw_body)
   - Retries:
     - If non-2xx, retry 3 times with exponential backoff (e.g., 1m, 5m, 15m)
   - Store delivery attempts and status.

DB tables:
- api_keys: id, user_id, key_hash, created_at, rotated_at, is_active
- webhook_endpoints: id, user_id, url, is_active, enabled_events(json), created_at
- webhook_events: id(uuid), user_id, event_type, payload_json, created_at
- webhook_deliveries: id, webhook_event_id, endpoint_id, attempt_number, status_code, response_body_excerpt, next_retry_at, delivered_at, created_at

UI:
- Event Log screen:
  - shows last 50 webhook events with status badges (Delivered / Failed / Retrying)
  - click into an event to view payload + delivery attempts

Security:
- Never store raw api keys after showing once. Store hash and only allow rotate/regenerate.
- Validate webhook URL format (https recommended).
