Build a production-quality “Publish Schedule” feature for the QuotePro web app.

This feature should allow an office/admin user to finalize the upcoming work schedule for cleaners, then click a “Publish Schedule” button that sends each cleaner their assigned weekly schedule automatically.

This is not just an email blast feature.
It should be a proper scheduling publication workflow with:
- draft vs published schedule states
- cleaner-specific schedule notifications
- versioning
- delivery tracking
- acknowledgment support
- change / republish support
- clean admin UX
- clean cleaner-facing email UX

PRODUCT GOAL

Once the office user finishes assigning cleaners/teams to jobs for the week, they click:
“Publish Schedule”

The system should:
1. create a published snapshot/version of the schedule
2. send each cleaner only their own assigned schedule for that week
3. format the schedule clearly in email
4. track delivery state
5. allow acknowledgment from cleaners
6. support future updates via republish logic
7. keep a history of published schedule versions

PRIMARY USERS

1. Office/Admin/User
- builds and manages weekly schedule
- assigns cleaners to jobs
- publishes schedule
- monitors which cleaners received / viewed / acknowledged their schedule
- republishes if assignments change

2. Cleaner / Team Member
- receives only their own schedule
- sees jobs by day
- sees key job details
- can acknowledge schedule
- can flag an issue if needed

WEB APP ONLY

Build this for the QuotePro web app.
Do not build iOS-specific implementation.
However, architect it in a way that could later be used by iOS or cleaner-facing mobile views.

CORE FEATURE NAME

Publish Schedule

HIGH-LEVEL FLOW

1. Admin creates/edits weekly schedule in draft state
2. Admin assigns cleaners or teams to jobs
3. Admin clicks “Publish Schedule”
4. System validates the schedule and shows publish preview
5. Admin confirms publish
6. System creates a schedule version snapshot
7. System sends each cleaner their personalized weekly schedule by email
8. System logs send status
9. Cleaners can acknowledge receipt
10. If schedule changes later, admin can click “Republish” and send only changed assignments to affected cleaners

REQUIRED CONCEPTS

Implement the following concepts in the system:

1. SCHEDULE STATUS
- draft
- published
- updated_after_publish

2. PUBLISHED VERSIONING
Every publish action should create a versioned snapshot, such as:
- Week of March 25–29, Version 1
- Week of March 25–29, Version 2

A version should preserve what was sent at that time.

3. CLEANER-SPECIFIC DELIVERY
Each cleaner should only receive:
- their own jobs
- their own dates/times
- their own notes
- their own teammate info where relevant

Do not send the entire company schedule to every cleaner.

4. ACKNOWLEDGMENT TRACKING
For each cleaner schedule notification:
- sent
- delivered if feasible
- opened/viewed if feasible
- acknowledged
- issue flagged

5. REPUBLISH / CHANGE HANDLING
If the admin changes the schedule after publish:
- mark schedule as updated_after_publish
- allow “Republish Schedule”
- show what changed
- send update only to affected cleaners if possible

FEATURE REQUIREMENTS

A. WEEKLY SCHEDULE MODEL

The weekly schedule should support:
- week start date
- week end date
- jobs/appointments
- customer name
- service address
- service date
- arrival time or time window
- service type
- assigned cleaner(s)
- assigned team
- notes / special instructions
- access instructions
- estimated duration if available
- internal notes
- cleaner-facing notes

Make sure cleaner-facing notes can be separated from internal office notes.

B. PUBLISH VALIDATION

Before publishing, validate the weekly schedule and surface issues such as:
- unassigned jobs
- jobs missing time/date
- cleaner missing email
- cleaner double-booked
- missing address/customer info
- cleaner overbooked / hours conflict if possible

Create a publish preview modal that clearly shows:
- total jobs in the selected week
- total cleaners being notified
- total assignments being sent
- warnings / conflicts
- whether publish can proceed
- option to review issues before sending

C. PUBLISH ACTION

When admin confirms publish:
- create a schedule publication record
- create schedule version snapshot
- generate cleaner-specific schedule summaries
- send cleaner-specific emails
- store send logs
- update schedule status to published

D. CLEANER EMAIL FORMAT

Create polished, readable cleaner schedule emails.
Do NOT send ugly table dumps.

Each cleaner email should contain:
- header with week range
- cleaner name
- grouped schedule by day
- for each job:
  - time / arrival window
  - customer name
  - address
  - service type
  - teammates if relevant
  - cleaner-facing notes
- weekly job count
- estimated weekly hours if available
- acknowledgment CTA/button/link
- link to view schedule online if feasible

Example structure:

Subject:
Your Schedule for March 25–29

Body:
Hi Maria,

Here is your schedule for the week of March 25–29.

Monday
8:00 AM — Smith Residence
123 Main St, Malvern, PA
Premium Clean
Teammate: Allison
Notes: Use side entrance. Dog in basement.

Tuesday
9:30 AM — Johnson Home
...

Summary
- 8 jobs this week
- Estimated hours: 31.5

Buttons:
- Acknowledge Schedule
- View Schedule

E. ACKNOWLEDGMENT FLOW

Cleaners should be able to acknowledge their schedule through:
- a secure link in email
- optional lightweight web page if needed

Support responses such as:
- Got it / Acknowledge
- I have an issue
- I’m unavailable

Track these responses in the admin interface.

F. REPUBLISH FLOW

If the schedule changes after publish:
- detect impacted cleaners
- show which jobs changed
- allow admin to republish
- support:
  1. notify affected cleaners only
  2. notify all cleaners again

For affected-cleaner updates, send an “updated schedule” email showing only what changed.

Example:
Subject:
Update to Your Schedule for Thursday

Body:
Your Thursday schedule has changed:
- Johnson Home moved from 10:00 AM to 11:30 AM
- New teammate: Ashley

Create an admin view for the published schedule showing:

- current version
- list of cleaners

- acknowledgment status per cleaner
- issue flagged status
