You are the senior engineer for QuotePro AI. Implement a “Seamless QuickBooks Online (QBO) Integration” that feels one-click for users.

PRODUCT GOAL
- User goes to Settings → QuickBooks → clicks “Connect QuickBooks”
- Completes Intuit OAuth
- After a quote is accepted, QuotePro can create:
  1) Customer (find-or-create)
  2) Invoice with line items
- UX must be frictionless: after initial connection, defaults take over.
- V1 is outbound only: QuotePro → QBO. No bi-directional sync, no payments, no reconciliation.

REFERENCES / REQUIRED BEHAVIOR
- Use Intuit OAuth 2.0 authorization code flow for QBO. (Official Intuit docs). 
- Use QBO Accounting API REST patterns: POST baseURL/v3/company/{realmId}/{resourceName}. 
- Create invoice workflow should follow Intuit’s invoice creation patterns (Line with SalesItemLineDetail).
- Refresh token expiration handling must exist (auto refresh; if refresh fails, prompt user to reconnect).
(Do not hardcode tokens; store securely; use refresh flow.)
These requirements align with Intuit documentation. 

========================
ARCHITECTURE
========================
Backend services needed:
1) OAuth connect + callback endpoints
2) Token storage (encrypted at rest)
3) QBO API client with:
   - automatic token refresh
   - retry/backoff for 429/5xx
   - idempotency protections on our side
4) Mapping layer:
   - QuotePro Quote -> QBO Customer + Invoice
5) QBO status UI + logs

Assume current stack:
- Node/TypeScript backend (or implement in the backend language already present)
- Existing DB available (Postgres or similar)
- Existing auth/userId

========================
OAUTH FLOW (INTUIT)
========================
Implement:
- GET /api/integrations/qbo/connect
  - Generates Intuit OAuth URL with state, redirect_uri, scopes
  - Stores state in DB/session with user_id
- GET /api/integrations/qbo/callback
  - Validate state
  - Exchange code for tokens (access + refresh)
  - Store encrypted tokens + realmId + scope
  - Set status=connected
- POST /api/integrations/qbo/disconnect
  - Marks disconnected_at, status=disconnected, clears tokens safely

Scopes:
- Use the minimum needed to create customers + invoices (accounting scope).
(Choose appropriate QBO scopes per Intuit’s scope model; keep minimal.)

Token refresh:
- Centralize in QBO client:
  - If access token expired or near-expiry, refresh using refresh token
  - Update DB with new tokens and expiry
  - If refresh fails due to invalid_grant, set status=needs_reauth and surface UI prompt to reconnect

IMPORTANT:
- NEVER log raw tokens.
- Encrypt tokens at rest.
- Use server-side secret management.

========================
QBO API CLIENT
========================
Implement a QBOClient that:
- Computes base URL per environment (sandbox vs production)
- Uses endpoints:
  - Create customer: POST /v3/company/{realmId}/customer
  - Query customer by email/displayname: GET /v3/company/{realmId}/query?query=...
  - Create item (optional): POST /v3/company/{realmId}/item
  - Create invoice: POST /v3/company/{realmId}/invoice
These follow QBO REST patterns where POST baseURL/v3/company/realmId/resourceName creates a resource.

Add:
- Retry policy for 429 and 5xx with exponential backoff (max 3 retries)
- Guardrail: if rate-limited, mark sync log with “retryable” and surface “Try again” UI

========================
QUOTE → QBO MAPPING RULES (SEAMLESS UX)
========================
On “Quote Accepted” event:
1) Ensure QBO connection exists and status=connected
2) Ensure a QBO Customer exists:
   - If qp_customer_id has mapping -> use qbo_customer_id
   - Else: query QBO for customer by email (preferred) or display name
   - If not found: create customer with:
       DisplayName, PrimaryEmailAddr, PrimaryPhone, BillAddr
   - Save mapping

3) Ensure line items map cleanly:
   V1 approach (fast + reliable):
   - Use ONE generic QBO Item if specific ones aren’t mapped yet:
       “Cleaning Services”
   - Put detail into Description (e.g., “Deep Clean + Oven + Fridge”)
   - Amount = Quote total, or split into multiple lines if desired
   Optional upgrade:
   - Create/find QBO Items per service type and save mappings

4) Create invoice:
   - CustomerRef = qbo_customer_id
   - Line[] using SalesItemLineDetail (or correct invoice line structure)
   - TxnDate = today (or service date if present)
   - PrivateNote includes QuotePro Quote ID and link back to QuotePro
   - EmailStatus can remain default (we are not sending invoice email in V1)

5) Store qbo_invoice_id in qbo_invoice_links to prevent duplicates.

Idempotency:
- If quote_id already has a qbo_invoice_link, do not create a new invoice.
- Provide UI option: “Create another invoice anyway” (advanced) but default OFF.
