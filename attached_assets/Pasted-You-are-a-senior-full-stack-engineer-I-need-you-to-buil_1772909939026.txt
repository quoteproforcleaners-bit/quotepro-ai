You are a senior full-stack engineer. I need you to build a production-ready V1 Jobber integration inside my existing app, QuotePro AI.

GOAL
Build the first version of a Jobber integration so QuotePro can:
1. Let a QuotePro user connect their Jobber account via OAuth
2. Store Jobber access + refresh tokens securely
3. Let the user choose whether accepted quotes should auto-create a Jobber client + request/job
4. Create the Jobber record automatically when a QuotePro quote is accepted
5. Show sync status and errors inside the QuotePro UI
6. Be architected cleanly so I can later add more Jobber workflows like estimate sync, follow-ups, and review requests

IMPORTANT PRODUCT STRATEGY
This integration should position QuotePro as the revenue layer on top of Jobber, not as a Jobber replacement.

TECHNICAL REQUIREMENTS
Use Jobber OAuth 2.0 flow and Jobber GraphQL API.
Use environment variables for all secrets.
Do not hardcode tokens or credentials.
Build this in a modular way with a dedicated Jobber integration service layer.

ASSUMPTIONS
- QuotePro already has authenticated users
- QuotePro already has quote records in the database
- There is already a quote acceptance event or quote status update that can trigger sync
- We can add DB fields/tables as needed
- We should support one Jobber connection per QuotePro account for V1

WHAT TO BUILD

1) DATABASE / DATA MODEL
Create whatever schema changes are needed, including something like:
- integrations_jobber
  - id
  - user_id or account_id
  - jobber_account_id
  - connected_by_jobber_user_id
  - access_token_encrypted
  - refresh_token_encrypted
  - access_token_expires_at
  - scopes
  - is_connected
  - auto_create_jobber_job_on_quote_accept (boolean)
  - default_sync_mode (enum: off, manual, automatic)
  - created_at
  - updated_at

- quote_jobber_syncs
  - id
  - quote_id
  - user_id or account_id
  - jobber_client_id
  - jobber_request_or_job_id
  - sync_status (pending, success, failed)
  - sync_trigger (manual, automatic)
  - error_message
  - raw_response_json
  - created_at
  - updated_at

If there is already an integrations table pattern in the codebase, follow that pattern instead of inventing a new one.

2) BACKEND / SERVER
Build these backend routes or server actions:

A. Start OAuth
- GET or POST /api/integrations/jobber/connect
- Generates Jobber OAuth authorization URL
- Includes secure state param
- Redirects user to Jobber OAuth page

B. OAuth Callback
- GET /api/integrations/jobber/callback
- Validates state
- Exchanges auth code for access + refresh tokens
- Encrypts and stores tokens
- Uses Jobber API to fetch the connected account and the user who installed the app
- Persists those IDs
- Marks integration connected
- Redirects back into QuotePro with success/failure notice

C. Integration Status
- GET /api/integrations/jobber/status
- Returns whether connected, connection metadata, sync preferences, and last sync info

D. Disconnect
- POST /api/integrations/jobber/disconnect
- Soft disconnects in QuotePro
- Clears or revokes local token usage safely
- Leaves audit trail

E. Update Preferences
- POST /api/integrations/jobber/preferences
- Save:
  - auto_create_jobber_job_on_quote_accept
  - default_sync_mode

F. Manual Sync Endpoint
- POST /api/integrations/jobber/sync-quote/:quoteId
- Creates the Jobber entities for a specific quote
- Returns success/failure plus created Jobber IDs

G. Automatic Sync Hook
- Hook into the quote acceptance flow
- When a quote becomes accepted:
  - check if Jobber connected
  - check if auto sync is enabled
  - if yes, create Jobber record automatically
  - save sync result
  - do not break quote acceptance if Jobber sync fails
  - surface non-blocking error to logs and UI

3) JOBBER SERVICE LAYER
Create a dedicated service module, for example:
- services/jobber/client.ts
- services/jobber/oauth.ts
- services/jobber/graphql.ts
- services/jobber/mappers.ts
- services/jobber/syncQuoteToJobber.ts

Responsibilities:

A. OAuth utilities
- Build authorization URL
- Exchange code for tokens
- Refresh access token when expired
- Centralize token refresh logic

B. GraphQL client
- Single reusable Jobber GraphQL request function
- Adds auth header
- Handles token refresh automatically
- Handles GraphQL errors gracefully
- Logs request IDs / errors safely

C. Mapping layer
Map a QuotePro quote into Jobber-compatible data:
- customer/client full name
- email
- phone
- address
- service title
- notes
- line items or summary
- quote total
- requested service date if available

D. Sync orchestration
- Check if client already exists in Jobber if feasible in V1; if not feasible, create client directly and document limitation
- Create the next logical Jobber object after client creation
- Save all returned external IDs
- Make sync idempotent:
  - prevent duplicate creation if the same accepted quote is synced twice
  - if already synced successfully, return existing sync result unless force=true

4) WHAT V1 SHOULD SYNC
Keep V1 narrow and useful.

When a quote is accepted in QuotePro, sync:
- Client / customer
- Property or address details if supported by available schema
- A service request, job, or closest appropriate Jobber work object
- Notes that indicate this came from QuotePro
- Quote total or pricing context in notes/description if direct field mapping is unclear

IMPORTANT:
Before coding, inspect the current Jobber GraphQL docs/schema integration points already available in the project or configuration and choose the correct object(s) for V1 creation.
Do not guess blindly if Jobber uses “request”, “job”, “client”, “property”, etc. Use the correct supported GraphQL mutations and types based on the implementation path you can verify in code comments/config/docs.
Abstract this well so we can swap request->job or add estimate sync later.
