URGENT: Fix a production bug in the QuotePro web app affecting customer-facing quote packages.

Current bug:
When quotes are sent to customers, the package options (for example Good / Better / Best) are rendering incorrectly.

Observed issues:
1. all package prices are the same, even though each package should have different pricing
2. package labels/descriptions are showing as “undefined”
3. customer-facing quote output is not correctly reflecting package differences
4. this is breaking trust in the quote experience and must be fixed ASAP

Treat this as a production-critical bug involving:
- pricing integrity
- quote rendering
- package mapping
- customer-facing output correctness

GOAL

Fix the quote generation and rendering flow so that:
- each package displays the correct package name/title
- each package displays the correct subtitle/description if applicable
- each package displays the correct package-specific price
- customer-facing quote pages/emails accurately reflect package differences
- no package ever shows “undefined”
- package pricing and labels are deterministic and validated before send/publish

LIKELY ROOT CAUSE AREAS TO AUDIT

Audit the entire quote package pipeline, especially:
1. package tier data model
2. package creation logic
3. package pricing calculation logic
4. package serialization / API response shaping
5. customer-facing quote renderer
6. fallback/default value behavior
7. mapping between package objects and UI components
8. package-specific adjustments, multipliers, add-ons, or inclusions
9. template rendering for quote email/page
10. data transformation between backend and frontend

I suspect one or more of these is happening:
- all packages are inheriting the same base total without package-specific deltas
- package-level pricing overrides are not being applied
- package metadata fields are null/undefined and not mapped correctly
- the frontend is reading the wrong field names
- the quote renderer is using fallback values for every package
- package description/title fields changed in schema but not in rendering
- package calculation objects are being cloned incorrectly or overwritten
- package tiers are generated but not independently recalculated

WHAT TO DO FIRST

Start by tracing one quote end-to-end:

1. inspect how Good / Better / Best packages are created in code
2. inspect the raw package data produced before rendering
3. inspect the API payload sent to the customer-facing quote page/email

   - title/description becomes undefined
   - pricing collapses to the same number across packages

Then fix the root cause, not just the symptom.

REQUIRED FIXES

1. PACKAGE TITLE / LABEL FIX
Ensure each package has a valid title/label.
Examples:
- Good
- Better
- Best
or whatever the app’s configured package names are

If there is also a subtitle/description, ensure it resolves correctly and never renders as undefined.

Implement safe fallbacks such as:
- if subtitle is missing, omit subtitle entirely
- never render literal “undefined” or “null” to customers

2. PACKAGE PRICE DIFFERENTIATION FIX
Ensure package-specific pricing is actually applied.

Each package should have its own computed amount based on:
- included services
- excluded services
- tier-specific add-ons
- package multipliers
- pricing rules
- manual adjustments
- upsell differences

If the intended behavior is:
- Good = base package
- Better = some added value/add-ons
- Best = premium package with more included work

then each should calculate independently and produce different totals when configured differently.

Do not allow all packages to silently reuse the same total unless they are truly identical by design.

3. CUSTOMER-FACING RENDERING FIX
Fix the quote page / email renderer so it correctly displays:
- package title
- package subtitle/description if present
- package amount
- package CTA button text

Examples:
- Accept Good
- Accept Better
- Accept Best

Do not render:
- “Good - undefined”
- “Accept Better - undefined”

If no secondary label exists, render only the package name cleanly.

4. VALIDATION BEFORE SEND
Before a quote is sent to a customer, validate package integrity.

Add validation rules such as:
- package title must exist
- package price must be numeric
- package list must not contain invalid undefined metadata
- package amounts should not all be identical if package contents differ
- if package generation fails, prevent sending and surface an admin error

This is very important.
Do not allow broken quote packages to be sent silently.

- generated package data

- missing title/description fields
